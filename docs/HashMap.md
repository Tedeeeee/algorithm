# 🧩 HashMap (해시맵)

HashMap은  
**Key–Value 형태로 데이터를 저장하는 자료구조**다.

코딩테스트에서 HashMap은  
👉 “자료를 저장한다”가 아니라  
👉 **‘빠르게 찾기 위해 구조를 바꾼다’**는 선택이다.

---

## 언제 HashMap을 떠올려야 하나?
아래 신호가 보이면 HashMap을 의심한다.

- 어떤 값이 있는지 빠르게 확인해야 한다
- 개수(빈도)를 세야 한다
- 값 ↔ 값의 대응 관계가 있다
- 입력 크기가 크다
- 정렬이 필요 없다

📌 핵심 신호
> **“배열로는 비효율적이다”**

---

## HashMap의 핵심 사고
HashMap의 본질은  
**“인덱스를 값으로 대체한다”**는 생각이다.

배열은
- 인덱스 → 값

HashMap은
- Key → 값

그래서 항상 이 질문을 던져야 한다.

> “이걸 배열 인덱스로 표현할 수 있을까?  
> 아니면 Key가 필요한가?”

---

## HashMap 사고 흐름 (공식처럼)

1. **Key는 무엇인가?**
    - 숫자?
    - 문자열?
    - 객체?
2. **Value는 무엇인가?**
    - 개수?
    - 인덱스?
    - 상태?
3. **조회 / 삽입 / 갱신 중 무엇이 잦은가?**

📌 이걸 명확히 하면  
👉 HashMap 사용 여부가 바로 결정된다.

---

## HashMap의 대표 사용 패턴

### 1️⃣ 빈도 세기 (Count)
- 숫자 카드
- 문자 개수
- 이름 등장 횟수

📌 Value = count

---

### 2️⃣ 존재 여부 확인
- 이미 등장했는가?
- 중복 체크

📌 Value = boolean or dummy 값

---

### 3️⃣ 매핑 / 대응 관계
- A → B 변환
- 이름 → 번호
- 번호 → 값

📌 양방향이면 두 개의 맵 고려

---

## HashMap 문제에서 반드시 점검할 것들
문제 풀기 전에 이걸 체크해라.

- Key가 유일한가?
- 값이 변경되는가?
- 정렬이 필요한가?
- Map 말고 배열이 더 빠르지 않은가?
- HashSet으로 충분하지 않은가?

---

## HashMap에서 자주 하는 실수
(진짜 많이 나온다)

- get() 결과 null 처리 안 함
- containsKey() 체크 누락
- Key 타입 혼동
- 불필요한 Map 사용
- 정렬이 필요한데 Map 사용

📌 HashMap은  
👉 **편하지만 함정도 많다.**

---

## HashMap vs HashSet
이 구분이 되면 깔끔해진다.

- 값의 “존재 여부”만 필요 → HashSet
- 값의 “개수 / 정보”가 필요 → HashMap

---

## 시간복잡도 감각
HashMap의 평균 시간복잡도:

- 삽입: O(1)
- 조회: O(1)
- 삭제: O(1)

📌 최악의 경우는 있지만  
👉 코테에서는 거의 고려하지 않는다.
