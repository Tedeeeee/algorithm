# 📦 Queue / Deque

큐(Queue)는  
**먼저 들어온 것이 먼저 나가는(FIFO) 자료구조**다.

코딩테스트에서 큐는  
👉 단순한 자료구조가 아니라  
👉 **“순서대로 처리한다”는 개념을 코드로 표현하는 도구**다.

스택이 “되돌아가기”라면,  
👉 큐는 **“기다렸다가 차례대로 처리하기”**다.

---

## 언제 큐를 떠올려야 하나?
문제에서 아래 신호가 보이면 큐를 의심한다.

- 순서대로 처리해야 한다
- 먼저 온 것부터 처리
- 대기열
- 차례차례 실행
- BFS
- 시뮬레이션

📌 핵심 신호
> **“먼저 들어온 애부터 처리해야 한다”**

---

## 큐의 핵심 사고
큐 문제의 본질은  
**“지금 처리할 것”과 “나중에 처리할 것”을 분리하는 사고**다.

그래서 항상 이 질문을 던져야 한다.

> “이건 지금 처리할 대상인가,  
> 아니면 줄 서서 기다려야 하나?”

---

## 큐 사고 흐름 (공식처럼 외우기)

1. **큐에 무엇을 넣을지 결정**
    - 값?
    - 인덱스?
    - 상태?
2. **언제 enqueue 하는지**
3. **언제 dequeue 하는지**
4. **처리 순서가 보장되는지**

📌 큐 문제는  
👉 **넣는 시점 / 빼는 시점**이 전부다.

---

## 큐의 대표 사용 패턴

### 1️⃣ BFS의 핵심 도구
- 큐 없이는 BFS 불가능
- “같은 거리 / 같은 시간” 단위 처리

📌 BFS = 큐 + 방문 처리

---

### 2️⃣ 시뮬레이션 / 대기열
- 프린터
- 작업 스케줄
- 순차 처리

📌 현실의 “줄 서기” 그대로

---

### 3️⃣ 원형 구조 / 회전
- 양쪽에서 삽입·삭제
- 원형 이동

👉 이때 **Deque** 사용

---

## Deque (덱)을 떠올려야 하는 순간
Deque는  
**앞과 뒤에서 모두 삽입/삭제가 가능한 큐**다.

아래 상황이면 일반 큐 ❌, Deque ⭕

- 앞에서도 꺼내고, 뒤에서도 꺼냄
- 회전한다
- 원형 이동
- 양쪽 포인터 이동

📌 억지로 인덱스로 구현하지 말고  
👉 **Deque를 써라**

---

## 큐 문제에서 반드시 점검할 것들
문제 풀기 전에 이 체크리스트를 훑어라.

- FIFO가 맞는가?
- 중간에 끼어드는 요소가 있는가?
- Deque가 필요한 상황은 아닌가?
- 큐에 중복 삽입될 가능성은 없는가?
- 종료 조건이 명확한가?

---

## 큐에서 자주 하는 실수
(실전에서 진짜 많이 터진다)

- 스택처럼 사용함
- poll / peek 혼동
- 큐 비었는데 poll
- 무한 루프
- Deque 써야 하는데 일반 큐 사용

📌 큐 문제의 오답은  
👉 **자료구조 선택 미스**가 많다.

---

## 큐 vs 스택 (감각 비교)

| 구분 | Queue | Stack |
|----|----|----|
| 처리 순서 | 먼저 온 것 | 가장 최근 |
| 핵심 개념 | 대기열 | 되돌리기 |
| 대표 사용 | BFS | DFS |
| 자료구조 | FIFO | LIFO |
