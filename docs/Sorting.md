# 🔃 Sorting (정렬)

정렬(Sorting)은  
**데이터를 특정 기준에 따라 재배치하는 것**이다.

코딩테스트에서 정렬은  
👉 그 자체가 목적이기보다는  
👉 **다른 알고리즘을 가능하게 만드는 준비 단계**다.

---

## 언제 정렬을 떠올려야 하나?
문제에서 아래 신호가 보이면 정렬을 의심한다.

- 크기 순서 비교
- 최소 / 최대를 반복적으로 찾는다
- 가까운 값 / 차이를 비교한다
- 두 값을 조합한다
- 이진 탐색이 필요하다
- 투 포인터를 쓰고 싶다

📌 핵심 신호
> **“순서가 정해지면 쉬워질 것 같다”**

---

## 정렬의 핵심 사고
정렬의 본질은  
**“순서를 강제로 만들어서 경우의 수를 줄인다”**는 생각이다.

정렬 전:
- 모든 경우를 다 봐야 함

정렬 후:
- 규칙적으로 건너뛸 수 있음
- 탐색 범위 축소 가능

📌 정렬은  
👉 **브루트포스를 정리하는 도구**다.

---

## 정렬 사고 흐름 (공식처럼)

1. **정렬이 필요한가?**
2. **정렬 기준은 무엇인가?**
    - 값
    - 길이
    - 문자열 규칙
3. **오름차순 / 내림차순?**
4. **안정성이 필요한가?**

📌 이걸 생각하고 정렬하면  
👉 이유 없는 sort가 사라진다.

---

## 정렬의 대표 사용 패턴

### 1️⃣ 기본 숫자 정렬
- 오름차순 / 내림차순
- 최솟값 / 최댓값 찾기

📌 대부분 내장 정렬로 충분

---

### 2️⃣ 커스텀 정렬 (Comparator)
- 조건 여러 개
- 우선순위 존재

📌 순서 중요:
1. 1차 기준
2. 2차 기준
3. 3차 기준

---

### 3️⃣ 정렬 + 탐색
- 이진 탐색
- 투 포인터

📌 정렬 안 하면 시작도 못 함

---

## 정렬 알고리즘은 얼마나 알아야 할까?
코딩테스트 기준:

- **구현 ❌**
- **개념 ⭕**

알아야 할 것:
- 평균 시간복잡도 O(N log N)
- 내장 정렬은 충분히 빠름
- Comparator 작성법

📌 버블 / 선택 / 삽입은  
👉 “왜 느린지”만 알면 된다.

---

## 정렬 문제에서 반드시 점검할 것들
문제 풀기 전에 이걸 체크해라.

- 정렬이 진짜 필요한가?
- 정렬 후 로직이 단순해지는가?
- 정렬 기준이 명확한가?
- 안정 정렬이 필요한가?
- 시간복잡도 괜찮은가?

---

## 정렬에서 자주 하는 실수
(실전에서 많이 터진다)

- 이유 없이 정렬
- 정렬 기준 누락
- Comparator 조건 순서 실수
- 문자열 비교 실수
- 정렬 후 인덱스 착각

📌 정렬은  
👉 **한 번만 실수해도 전체가 무너진다.**

---

## 정렬과 다른 알고리즘의 연결
정렬이 들어가면 흐름이 이렇게 이어진다.

- 정렬 + 투 포인터
- 정렬 + 이진 탐색
- 정렬 + 그리디
- 정렬 + 스택 (단조 스택)

📌 정렬은  
👉 **연결 허브**다.
