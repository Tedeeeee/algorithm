# 🟢 Greedy (그리디)

그리디(Greedy)는  
**매 순간 가장 좋아 보이는 선택을 하는 알고리즘 설계 기법**이다.

코딩테스트에서 그리디는  
👉 “대충 최선”이 아니라  
👉 **“이 선택이 항상 옳다는 걸 증명할 수 있을 때만 쓰는 방법”**이다.

---

## 언제 그리디를 떠올려야 하나?
문제에서 아래 신호가 보이면 그리디를 의심한다.

- 매 단계에서 선택해야 한다
- 한 번 선택하면 되돌릴 수 없다
- 전체를 보지 않고도 결정 가능해 보인다
- 최댓값 / 최솟값을 만든다
- 정렬 후 순차 처리

📌 핵심 신호
> **“지금 이 선택이 나중에 영향을 안 줄 것 같다”**

---

## 그리디의 핵심 사고
그리디의 본질은  
**“지금의 최선 선택이 전체 최선으로 이어진다”**는 성질이다.

그래서 항상 이 질문을 던져야 한다.

> “이 선택을 지금 해도  
> 나중에 후회하지 않는가?”

이 질문에 **논리적으로 YES**가 아니면  
👉 그리디는 쓰면 안 된다.

---

## 그리디 사고 흐름 (공식처럼)

1. **선택 기준 정의**
    - 무엇을 기준으로 고를 것인가?
2. **정렬 여부 판단**
3. **선택 후 상태 변화 분석**
4. **다음 선택에 영향 없는지 확인**

📌 증명이 안 되면  
👉 그리디 ❌

---

## 그리디의 대표 패턴

### 1️⃣ 정렬 + 순차 선택
- 가장 흔한 그리디 형태
- 정렬이 핵심

📌 예시 상황
- 최소 비용
- 최대 개수
- 빠른 종료

---

### 2️⃣ 가장 작은 것 / 가장 큰 것 선택
- 국소 최적 선택
- 전체 최적 보장

📌 항상 성립하지는 않음 → 증명 필요

---

### 3️⃣ 교환 논증 (Exchange Argument)
- 선택을 바꿔도 결과가 나빠지지 않음을 증명

📌 이게 가능하면 그리디 OK

---

## 그리디 문제에서 반드시 점검할 것들
문제 풀기 전에 이걸 체크해라.

- 선택을 되돌릴 필요 없는가?
- 정렬 기준이 명확한가?
- 선택 순서를 바꿔도 결과가 같은가?
- DP로 풀어야 하는 건 아닌가?

---

## 그리디에서 자주 하는 실수
(진짜 많이 틀린다)

- 근거 없는 선택
- 정렬 기준 잘못 설정
- 국소 최적이 전체 최적이라고 착각
- DP 문제를 그리디로 풀려고 함
- 반례 하나에 무너짐

📌 그리디는  
👉 **반례에 가장 취약한 알고리즘**이다.

---

## 그리디 vs 다른 알고리즘
이 구분이 되면 실수가 줄어든다.

- 선택을 되돌릴 수 있음 → DP
- 모든 경우 → 브루트포스
- 정렬 후 두 값 → 투 포인터
- 범위 줄이기 → 이진 탐색
